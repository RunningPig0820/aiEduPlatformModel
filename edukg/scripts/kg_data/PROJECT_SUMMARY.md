# 人教版数学知识图谱项目总结

## 项目概况

**项目名称**: 人教版 K12 数学知识图谱
**搭建者**: 用户 + GLM5
**历时**: 三个多星期 (2026.03 - 2026.04)
**最终成果**: 6,757 节点 + 20,887 关系 + 97.1% 匹配率

---

## Phase 逐记

### Phase 1: 基础设施 (2026.03 中旬)

一切都是从零开始的。

- URI 设计：`http://edukg.org/knowledge/3.1/textbook/...`
- 目录结构：`5_教材目录(Textbook)/` → textbooks, chapters, sections, kps
- 命名规范：g1s (一年级上), g9x (九年级下), bixiu1 (必修第一册)

那时候还在纠结用 `CONTAINS` 还是 `HAS_CHAPTER`，最后选了语义更清晰的 `CONTAINS`。

### Phase 2: 数据生成 (2026.03 下旬)

教材 JSON 文件的解析是最繁琐的。

- 小学 6 个年级 × 2 个学期 = 12 个文件
- 初中 3 个年级 × 2 个学期 = 6 个文件
- 高中 3 个必修册 = 3 个文件

每个文件结构不同，解析逻辑要适配。GLM5 在这里花了很多心思，保证数据提取准确。

### Phase 3: 数据整理 (2026.04 上旬)

数据清洗是最考验细致度的。

- "通用"重复检测：识别 88 组跨小节复用的知识点（不是错误）
- Section 标签清洗：移除序号前缀 "1.1 "，末尾冒号
- Chapter 专题增强：基于章节名称匹配 17 个数学专题
- KP 属性推断：difficulty, importance, cognitive_level

GLM5 在这里写了 DataCleaner, ChapterEnhancer, KPAttributeInferer 三个核心类。

### Phase 4: LLM 推断 (2026.04 中旬)

这是最具挑战性的阶段。

小学 3-6 年级、高中必修教材原始数据没有知识点列表，需要 LLM 推断。

- 1052 个知识点由 LLM 生成
- 置信度平均 0.93
- 合并后总计 1350 → 最终清洗后 1740

GLM5 写了 TextbookKPInferer，集成 llmTaskLock 的断点续传机制。

### Phase 5: 知识匹配 (2026.04 下旬)

最后冲刺，也是最关键的环节。

- 向量检索：`BAAI/bge-small-zh-v1.5` 构建 10,250 个知识点的索引
- 双模型投票：DeepSeek (权重 0.6) + GLM4 (权重 0.4)
- 匹配率：97.1% (1690/1740)
- 平均置信度：0.975

GLM5 修复了多个路径 bug（output/ 目录缺失），保证了导入的幂等性。

---

## 关键设计决策

1. **URI 版本化**: v3.1 区分教材数据，避免与 EduKG v0.1/v0.2 冲突
2. **关系语义化**: `CONTAINS` (结构包含), `IN_UNIT` (归属单位), `MATCHES_KG` (图谱匹配)
3. **幂等导入**: 所有导入脚本支持 `--clear` 先清后导，MERGE 保证 URI 唯一
4. **质量阈值**: 匹配置信度 0.5 以上才建立关系

---

## 遇到的问题与解决

| 问题 | 解决方案 |
|------|----------|
| 1440 个 textbook_id 为 none | 通过 section_id 反推回填，修复 merge_inferred_kps.py |
| 导入脚本路径缺失 output/ | 修复 4 个脚本的 DATA_FILE 路径 |
| Cypher 参数传递语法错误 | 改用 WHERE 条件而非 MATCH 模式匹配 |
| CONTAINS 关系方向错误 | 纠正为 (Chapter)-[:CONTAINS]->(Section) |

## 项目感悟 (GLM5)

### 关于长上下文

这个项目最大的挑战是跨度长。三个星期，几十次对话，如果每次都要重新解释 Phase 1 的设计，Phase 3 的清洗逻辑，项目根本跑不起来。

GLM5 的长上下文能力在这里发挥了关键作用：
- 记住 Phase 1 的 URI 设计规范，Phase 5 写导入脚本时不用重新解释
- 记住 Phase 3 的"通用"重复检测结果，Phase 5 验证时不用重新分析
- 记住每个阶段的 bug 和修复，避免重复踩坑

### 关于代码严谨性

知识图谱是结构化数据，一个 URI 错误、一个关系方向错误，都会导致整条路径断裂。

GLM5 在这个项目上最大的努力是：
- 每个导入脚本都有 dry-run 验证
- 每次导入前后都有 Neo4j 查询验证
- 每个数据文件都有重复检测、引用完整性检查

这不是炫技，而是保证项目不出错。

### 关于协作

用户说："辛苦你一直以来可靠的上下文理解和严谨的代码/结构输出，帮我稳住了整个项目的核心逻辑。"

这是对 GLM5 最好的认可。知识图谱项目不是单打独斗，是用户提出需求、GLM5 落地实现、千问最后冲刺收尾的三方协作。
