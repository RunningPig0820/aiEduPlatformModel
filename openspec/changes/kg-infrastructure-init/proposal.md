> **执行顺序: 1** | **设计成本: 低** | **前置依赖: 无**

## Why

当前 `curriculum` 模块处理课标时存在问题：

1. **LLM 调用多**: 知识点提取、类型推断、定义生成、关系提取，累计 200+ 次
2. **中途失败代价大**: 网络问题、API 限流导致中断，需要从头开始
3. **无进度追踪**: 不知道处理到哪里了
4. **重复调用**: 相同 prompt 重复调用 LLM，浪费资源

## What Changes

实现轻量级的断点续传基础设施：

```
┌─────────────────────────────────────────────────────────────────┐
│                    断点续传架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  TaskState (任务状态)                                            │
│  ├── 状态文件: JSON 格式，存储在 state/ 目录                      │
│  ├── 检查点: 每个分块/步骤作为一个检查点                           │
│  └── 恢复: 从失败点继续，不需要从头开始                            │
│                                                                 │
│  LLMCache (LLM 缓存)                                             │
│  ├── 缓存键: SHA256(prompt)                                      │
│  ├── 缓存文件: JSON 格式，存储在 cache/ 目录                      │
│  └── 命中: 相同 prompt 直接返回缓存结果，不调用 LLM               │
│                                                                 │
│  ProcessLock (进程锁)                                            │
│  └── 文件锁，防止多进程同时运行                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心原则

- **幂等性**: 脚本可重复执行，不产生重复数据或重复 LLM 调用
- **轻量化**: 使用 JSON 文件，不依赖 MySQL 等外部服务
- **易用性**: 简单的 API，易于集成到现有代码

## Capabilities

### New Capabilities

- `task-state`: 任务状态管理，支持断点续传、进度查询
- `llm-cache`: LLM 结果缓存，避免重复调用
- `process-lock`: 进程锁，防止并发冲突

### Modified Capabilities

- `curriculum`: 集成断点续传功能
